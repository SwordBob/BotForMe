## 第 5 章 信息系统工程 ⭐⭐⭐

### 5.1 软件工程

**核心内容：** 软件开发生命周期、开发方法、需求工程

#### 重点知识点：

**1. 软件生命周期模型**

| 模型 | 特点 | 适用场景 |
|------|------|----------|
| 瀑布模型 | 顺序执行，阶段清晰 | 需求明确、技术成熟 |
| 增量模型 | 分批交付，降低风险 | 需要快速交付部分功能 |
| 螺旋模型 | 风险驱动，迭代开发 | 大型复杂项目 |
| 敏捷开发 | 迭代增量，快速响应变化 | 需求变化频繁 |

**2. 需求工程** ⭐⭐⭐⭐⭐

```
需求获取 → 需求分析 → 需求定义 → 需求验证
```

- **需求跟踪矩阵（RTM）**：追踪需求到交付，确保每个需求都被实现

**3. 软件测试**

| 测试类型 | 执行者 | 目的 |
|----------|--------|------|
| 单元测试 | 开发人员 | 测试最小可测试单元 |
| 集成测试 | 测试人员 | 测试模块间接口 |
| 系统测试 | 测试人员 | 测试整个系统 |
| 验收测试 | 客户 | 确认是否满足需求 |

- **黑盒测试**：关注输入输出，不关心内部实现
- **白盒测试**：关注内部逻辑和代码路径

### 5.2 数据工程

- 数据架构设计
- 数据库设计范式（1NF、2NF、3NF）
- 数据仓库与大数据

### 5.3 系统集成

- 集成技术：EAI（企业应用集成）、ESB（企业服务总线）、SOA（面向服务架构）
- 接口设计标准

### 5.4 安全工程

**信息安全三要素（CIA）：**
- 机密性（Confidentiality）
- 完整性（Integrity）
- 可用性（Availability）

- 安全等级保护制度

## 第 7 章 项目立项管理 ⭐⭐⭐⭐

### 7.1 可行性研究 ⭐⭐⭐⭐⭐ **案例分析必考**

**四个方面：**
1. **技术可行性**：技术能否实现
2. **经济可行性**：投资回报率（ROI）、净现值（NPV）、内部收益率（IRR）
3. **法律可行性**：合规性
4. **操作可行性**：组织能否接受

#### 📐 计算题公式（必背）：

**投资回收期：**
```
投资回收期 = 累计净现金流量开始出现正值的年份 - 1 + (上年累计净现金流量的绝对值 / 当年净现金流量)
```

**净现值 NPV：**
```
NPV = Σ(净现金流量 / (1 + 折现率)^t)
```
- NPV > 0：项目可行
- NPV < 0：项目不可行

**投资回报率 ROI：**
```
ROI = (年平均利润 / 总投资) × 100%
```

### 7.2 项目评估

- 初步可行性研究 vs 详细可行性研究
- 项目论证：技术、经济、环境、社会

### 7.3 项目章程 ⭐⭐⭐⭐⭐

**项目章程的作用：**
- 正式批准项目
- 授权项目经理

**项目章程内容（记忆要点）：**

### 7.4 招投标管理

**工具：** 专家判断、数据收集、人际关系技能

**内容：** 所有子管理计划的整合
- 范围管理计划、进度管理计划、成本管理计划
- 质量管理计划、资源管理计划、沟通管理计划
- 风险管理计划、采购管理计划、干系人参与计划

### 8.3 指导与管理项目工作

- 执行计划中的活动
- 创建可交付成果
- 管理项目知识

### 8.4 管理项目知识

- 知识分享与转移
- 经验教训登记册

### 8.5 监控项目工作 ⭐

- 跟踪、审查、报告项目进展
- 实施变更请求

### 8.6 实施整体变更控制 ⭐⭐⭐⭐⭐ **案例分析高频考点**

#### 变更控制流程（必背）：
```
1. 提出变更请求
2. 评估变更影响（范围、进度、成本、质量）
3. 提交变更控制委员会（CCB）审批
4. 批准则更新计划，拒绝则记录原因
5. 实施变更
6. 验证变更效果
```

**关键要点：**
- ✅ 所有变更必须书面记录
- ✅ CCB 是决策机构，不是执行机构
- ✅ 变更实施后必须验证
- ✅ 项目经理负责评估影响，CCB 负责审批

### 8.7 结束项目或阶段

- 验收可交付成果
- 归档项目文件
- 总结经验教训
- 释放资源

- 100% 规则（包含所有工作）

项目 → 阶段/可交付成果 → 工作包 → 活动

## 第 10 章 项目进度管理 ⭐⭐⭐⭐⭐ **计算题核心**

### 10.1 规划进度管理

**输出：** 进度管理计划

### 10.2 定义活动

**工具：** 分解、滚动式规划  
**输出：** 活动清单、活动属性、里程碑清单

### 10.3 排列活动顺序

**依赖关系类型：**
- 强制性依赖（硬逻辑）
- 选择性依赖（软逻辑）
- 外部依赖
- 内部依赖

**逻辑关系：**
- 完成 - 开始（FS）最常见
- 开始 - 开始（SS）
- 完成 - 完成（FF）
- 开始 - 完成（SF）少见

**输出：** 项目进度网络图

### 10.4 估算活动持续时间

#### 估算方法：

| 方法 | 特点 | 准确度 |
|------|------|--------|
| 类比估算 | 自上而下，快速 | 低 |
| 参数估算 | 基于历史数据 | 中 |
| 三点估算 ⭐⭐⭐ | 考虑风险，准确 | 高 |

#### 三点估算公式（必背）：
```
期望时间 Te = (乐观 + 4×最可能 + 悲观) / 6

标准差 σ = (悲观 - 乐观) / 6

方差 σ² = [(悲观 - 乐观) / 6]²
```

### 10.5 制定进度计划 ⭐⭐⭐⭐⭐ **计算题必考**

#### 关键路径法 CPM：

```
关键路径：网络图中最长的路径，决定项目最短工期
关键路径上的活动总时差 = 0
```

#### 六个时间参数（必背）：

| 参数 | 含义 | 计算公式 |
|------|------|----------|
| ES | 最早开始时间 | 正推法，取紧前 EF 最大值 |
| EF | 最早结束时间 | EF = ES + 持续时间 |
| LS | 最晚开始时间 | LS = LF - 持续时间 |
| LF | 最晚结束时间 | 逆推法，取紧后 LS 最小值 |
| TF | 总时差 | TF = LS - ES = LF - EF |
| FF | 自由时差 | FF = 紧后活动 ES 的最小值 - 本活动 EF |

#### 计算规则：

**正推法（求 ES、EF）：**
- 从左到右计算
- 多个紧前活动时，ES 取最大值

**逆推法（求 LS、LF）：**
- 从右到左计算
- 多个紧后活动时，LF 取最小值

#### 资源优化：

| 技术 | 特点 | 影响 |
|------|------|------|
| 资源平衡 | 解决资源冲突 | 可能改变关键路径 |
| 资源平滑 | 在时差内调整 | 不改变关键路径 |

#### 进度压缩技术：

| 技术 | 方法 | 影响 |
|------|------|------|
| 赶工 | 增加资源 | 增加成本，不增加风险 |
| 快速跟进 | 并行进行 | 增加风险，不增加成本 |

### 10.6 控制进度

- 监督进度状态
- 更新进度模型
- 管理进度变更

## ⚠️ 易错点提醒

### 1. 范围确认 vs 质量控制
| 对比项 | 范围确认 | 质量控制 |
|--------|----------|----------|
| 执行者 | 客户/发起人 | 项目团队 |
| 目的 | 正式验收 | 检查质量 |
| 时间 | 阶段结束时 | 执行过程中 |

### 2. 关键路径上的活动时差
- **总时差 = 0**（不是自由时差）
- 关键路径可能有多条
- 关键路径可能变化

### 3. 变更请求的审批
- **CCB 审批**，不是项目经理
- 项目经理负责评估影响
- 所有变更必须书面记录

### 4. WBS 分解
- **工作包是最底层**，不是活动
- WBS 词典包含工作包的详细信息
- WBS 不包含活动顺序

### 5. 三点估算公式
- Te = (O + 4M + P) / 6，**不要记错系数**
- 标准差 σ = (P - O) / 6
- 概率计算：±1σ = 68.26%，±2σ = 95.46%，±3σ = 99.73%

### 6. 进度压缩
- **赶工**：增加资源 → 增加成本
- **快速跟进**：并行进行 → 增加风险
- 两者都可能改变关键路径
