## 📖 第一部分：软件工程（重点！）

### 1.1 软件工程基础 ⭐⭐⭐

#### 定义与三要素
- **IEEE定义**：将系统的、规范的、可度量的工程化方法应用于软件开发、运行和维护的全过程
- **三要素**：
  ```
  1. 方法 - 技术手段（支持全生命周期）
  2. 工具 - 自动/半自动生成文档、支持管理
  3. 过程 - 质量、进度、成本控制（含配置管理、QA）
  ```

#### 考试要点
- ✅ 记住IEEE定义的关键词：**系统、规范、可度量**
- ✅ 三要素的作用和区别
- ✅ 过程包括：人员组织、计划跟踪、成本估算、质量保证、配置管理

### 1.2 软件架构设计 ⭐⭐⭐⭐⭐（超高频考点）

#### 架构风格分类（Garlan & Shaw）
```
┌─────────────────────────────────────────────┐
│ 1. 数据流风格                                  │
│    - 批处理序列                                │
│    - 管道/过滤器 ★                            │
├─────────────────────────────────────────────┤
│ 2. 调用/返回风格                               │
│    - 主程序/子程序                            │
│    - 面向对象 ★                              │
│    - 层次结构                                 │
├─────────────────────────────────────────────┤
│ 3. 独立构件风格                                │
│    - 进程通信                                 │
│    - 事件驱动 ★（消息驱动系统）                │
├─────────────────────────────────────────────┤
│ 4. 虚拟机风格                                  │
│    - 解释器                                   │
│    - 基于规则 ★（AI系统）                      │
├─────────────────────────────────────────────┤
│ 5. 仓库风格                                    │
│    - 数据库系统 ★                            │
│    - 黑板系统（知识密集型）                    │
│    - 超文本系统                               │
└─────────────────────────────────────────────┘
```

#### 架构评估（必考！）
- **核心概念**：
  - **敏感点**：影响**单个**质量属性的特性
  - **权衡点**：影响**多个**质量属性的特性 ⭐
    - 例：加密级别↑ → 安全性↑ 但 性能↓
  
- **评估方法**：
  | 方法 | 全称 | 特点 | 使用场景 |
  |------|------|------|----------|
  | ATAM | 架构权衡分析法 | 基于场景，最常用 | 多质量属性权衡 |
  | SAAM | 软件架构分析法 | 最早的方法 | 质量属性分析 |
  | CBAM | 成本效益分析法 | 考虑成本 | 经济决策 |

- **场景三要素**：
  ```
  刺激 (Stimulus) → 环境 (Environment) → 响应 (Response)
  ```

#### 考题示例
```
【选择题】某系统需要支持高并发和快速响应，架构师决定采用事件驱动架构。
这属于哪种架构风格？
A. 数据流风格  B. 调用/返回风格  C. 独立构件风格  D. 仓库风格
答案：C

【选择题】加密算法的选择会影响系统的安全性和性能，这个决策点称为？
A. 敏感点  B. 权衡点  C. 风险点  D. 关键点
答案：B
```

### 1.3 需求工程 ⭐⭐⭐⭐

#### 需求层次
```
业务需求（组织目标）
    ↓
用户需求（用户任务）
    ↓
系统需求（功能+非功能）
```

#### QFD（质量功能部署）
- **三种需求**：
  1. **常规需求**：基本功能
  2. **期望需求**：用户明确要求的
  3. **意外需求**：超出用户期望的（兴奋点）

#### 需求过程
```
需求获取 → 需求分析 → 需求规格说明书(SRS) → 需求验证与确认
```

#### 分析方法对比
| 方法 | 核心模型 | 工具 | 特点 |
|------|----------|------|------|
| **结构化分析(SA)** | 数据字典 | DFD、E-R图、STD | 自顶向下 |
| **面向对象分析(OOA)** | 用例模型 | 类图、交互图 | 对象为中心 |

#### 用例关系（重点！）
- **包含(Include)**：提取公共行为，**必须执行**
- **扩展(Extend)**：分支场景，**条件执行**
- **泛化(Generalization)**：父子用例，继承关系

### 1.4 软件设计 ⭐⭐⭐⭐

#### 结构化设计(SD)
- **核心原则**：**高内聚、低耦合**
- **阶段**：
  1. 概要设计 → 模块结构图
  2. 详细设计 → 过程设计

#### 面向对象设计(OOD) - 设计原则（必背！）
```
SOLID 原则：
┌─────────────────────────────────────┐
│ S - 单一职责原则                      │
│    一个类只负责一个职责               │
├─────────────────────────────────────┤
│ O - 开闭原则 ★                       │
│    对扩展开放，对修改封闭             │
├─────────────────────────────────────┤
│ L - 里氏替换原则                     │
│    子类必须能替换父类                │
├─────────────────────────────────────┤
│ I - 接口隔离原则                     │
│    使用多个专门接口优于单一总接口     │
├─────────────────────────────────────┤
│ D - 依赖倒置原则                     │
│    依赖抽象而非具体实现              │
└─────────────────────────────────────┘

其他原则：
- 组合重用原则：优先使用组合而非继承
- 迪米特法则（最少知识原则）：一个对象对其他对象了解最少
```

#### 设计模式分类
| 类型 | 数量 | 常见模式 |
|------|------|----------|
| **创建型** | 5种 | 工厂、抽象工厂、单例、原型、建造者 |
| **结构型** | 7种 | 适配器、桥接、装饰、外观、代理、组合、享元 |
| **行为型** | 11种 | 观察者、策略、命令、状态、模板方法等 |

### 1.5 软件实现与测试 ⭐⭐⭐⭐⭐（计算题重点）

#### 配置管理活动
```
配置计划 → 配置标识 → 配置控制 → 状态记录 → 配置审计 → 发布管理与交付
```

#### 测试分类
```
软件测试
├── 静态测试（不运行程序）
│   ├── 桌前检查
│   ├── 代码走查
│   └── 代码审查
│
└── 动态测试（运行程序）
    ├── 白盒测试（结构测试）★
    │   └── 关注内部逻辑
    │
    └── 黑盒测试（功能测试）★
        └── 关注输入输出
```

#### 白盒测试覆盖标准（必考！强度递增）
```
语句覆盖 < 判定覆盖 < 条件覆盖 < 条件/判定覆盖 < 条件组合覆盖 < 路径覆盖
  最弱                                              最强
```

| 覆盖标准 | 要求 | 强度 |
|----------|------|------|
| **语句覆盖** | 每条语句至少执行一次 | ⭐ 最弱 |
| **判定覆盖** | 每个分支的真/假至少一次 | ⭐⭐ |
| **条件覆盖** | 每个条件的真/假至少一次 | ⭐⭐⭐ |
| **条件/判定覆盖** | 同时满足判定和条件覆盖 | ⭐⭐⭐⭐ |
| **条件组合覆盖** | 所有条件组合至少一次 | ⭐⭐⭐⭐⭐ |
| **路径覆盖** | 所有可能路径至少一次 | ⭐⭐⭐⭐⭐⭐ 最强 |

#### 黑盒测试方法
| 方法 | 适用场景 | 特点 |
|------|----------|------|
| **等价类划分** | 输入数据分类 | 有效/无效等价类 |
| **边界值分析** ★ | 输入边界 | 错误多发生在边界 |
| **判定表** | 多条件组合 | 条件桩+动作桩 |
| **因果图** | 复杂逻辑 | 输入输出关系 |
| **正交试验法** | 多因素多水平 | 减少测试用例 |

#### 考题示例
```
【选择题】以下哪种测试覆盖标准的覆盖强度最强？
A. 语句覆盖  B. 判定覆盖  C. 条件覆盖  D. 路径覆盖
答案：D

【选择题】测试人员设计测试用例时，重点关注输入数据的边界值，
这属于哪种测试方法？
A. 等价类划分  B. 边界值分析  C. 判定表  D. 因果图
答案：B

【计算题】给定以下代码段，至少需要多少个测试用例才能实现判定覆盖？
if (A > 1 and B = 0) then
  X = X / A
end if
if (A = 2 or X > 1) then
  X = X + 1
end if

解题思路：
- 判定覆盖要求每个判定的真/假至少执行一次
- 第一个判定：(A>1 and B=0) 需要 真1次 + 假1次
- 第二个判定：(A=2 or X>1) 需要 真1次 + 假1次
- 最少需要 2 个测试用例（精心设计可同时满足）
```

### 1.6 部署与交付（新趋势）⭐⭐⭐

#### 持续交付 vs 持续部署
| 概念 | 定义 | 特点 |
|------|------|------|
| **持续交付** | 自动化发布过程 | 一键部署到生产环境 |
| **持续部署** | 自动通过测试后自动部署 | 无需人工干预 |

#### 部署策略（重点！）
| 策略 | 方法 | 优点 | 缺点 |
|------|------|------|------|
| **蓝绿部署** | 新旧版本同时运行，切换流量 | 快速回滚、零停机 | 资源成本高 |
| **金丝雀部署** | 灰度发布，少量用户先试 | 风险可控 | 部署时间长 |

#### 容器技术
- **Docker**：应用打包和隔离
- **Kubernetes(K8s)**：容器编排
- **原则**：不可变服务器、统一镜像仓库

### 1.7 软件过程能力成熟度模型（CSMM）⭐⭐⭐⭐⭐

#### 4个能力域
```
┌──────────────────────────────────────────────┐
│ 1. 治理                                       │
│ 2. 开发与交付                                  │
│ 3. 管理与支持                                  │
│ 4. 组织管理                                    │
└──────────────────────────────────────────────┘
```

#### 5个成熟度等级（必背！）
| 等级 | 名称 | 特征 | 关键词 |
|------|------|------|--------|
| **1级** | 初始级 | 过程不确定，依赖个人 | 无序、混乱 |
| **2级** | 项目规范级 | 项目按计划，有基本规范 | 项目级、规范化 |
| **3级** | 组织改进级 | 组织范围内稳定实现 | **组织标准**、稳定 |
| **4级** | 量化提升级 | 使用统计分析技术 | **量化**、统计 |
| **5级** | 创新引领级 | 技术创新，引领行业 | **创新**、引领 |

#### 记忆口诀
```
一初无序靠个人
二项规范有标准
三组稳定成体系
四量统计精管理
五新创新领行业
```

#### 考题示例
```
【选择题】某组织已经建立了标准的过程体系，并在组织范围内推广使用，
这属于CSMM的哪个等级？
A. 2级  B. 3级  C. 4级  D. 5级
答案：B

【选择题】在CSMM中，开始使用统计分析技术进行量化管理的等级是？
A. 2级  B. 3级  C. 4级  D. 5级
答案：C
```

## 📖 第二部分：数据工程

### 2.1 数据建模 ⭐⭐⭐⭐

#### 三类模型
```
现实世界
    ↓ (抽象)
概念模型（信息世界）← 用户观点，E-R图
    ↓ (转换)
逻辑模型（机器世界）← 数据结构，关系模型
    ↓ (实现)
物理模型（计算机世界）← 具体实现，表/索引
```

#### 概念模型
- **元素**：实体、属性、域、键、关联
- **要求**：真实反映现实、简洁易理解、易于变动、易于转换
- **工具**：E-R图（实体-关系图）

#### 逻辑模型
- **关系模型对应**：
  - 实体 → 关系（表）
  - 属性 → 属性（列）
  - 联系 → **外键**

- **完整性约束**（必考！）：
  1. **实体完整性**：主键非空且唯一 ⭐
  2. **参照完整性**：外键必须引用有效主键 ⭐
  3. **用户定义完整性**：业务规则约束

#### 物理模型
- **内容**：表、字段、索引、存储过程、触发器
- **考虑**：性能优化，可能进行**反规范化**

### 2.2 数据建模过程
```
数据需求分析 → 概念模型设计 → 逻辑模型设计 → 物理模型设计
```
